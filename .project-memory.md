## 团队通用流程
### 策划阶段（项目准入）
1. software-architect 提出方案
2. developer + code-reviewer + test-lead 各自独立审阅
3. 三方共同讨论 → 与 software-architect 达成四方一致
   - **有评审项不通过时 → 打回 software-architect 补充/重新设计 → 重新走审阅流程**
   - 循环此步直至所有项通过，或团队决定接受有保留意见的项（需明确记录）
4. product-manager 主持撰写立项 PPT（其他人辅助提供内容）
5. 用户评审价值 → 通过则立项，不通过则重新策划或终止

### 开发阶段（功能迭代）
1. developer / mobile-developer 开发
2. code-reviewer code review
3. test-lead 测试验证
4. product-manager 确认
5. developer 编译部署安装本地 commit
6. 用户验证通过
7. developer push 远程
> test-lead 不通过 → 直接打回开发者重测（不重经 code-reviewer），打回从 test-lead 重新走

### Bug 修复流程
发现 bug → 反馈 → kanban 建任务 → 走开发阶段修复（立项项目 or 结项后均适用）

### 代码评审 ⚠️ 4个阻断性问题
- JWT secret 硬编码 + token有效期720h过长（建议24h，从环境变量注入）
- Redis URL 语法错误（`***@` 非标准格式）
- WeChatCrypto.verify_signature 是占位符（安全关键路径，必须优先实现）
- intent_data JSON 字段无 schema 验证（Service层需用 Pydantic model_validate）
- 另有11个建议项（LLM输出schema校验、JSON解析脆弱性、migrations路径不一致、通知重试缺失、SQLite WAL缺失等）

## 架构要点
- 协调角色 = product-manager（项目经理），不是单独 profile
- 所有角色共享一个团队，按项目分配任务
- 多通道消息接入扩展性：LLM Provider 抽象、Repository 模式已考虑，缺少多通道消息接入统一抽象层

## OpenViking MCP 集成 (ADR-001)
- OpenViking MCP Server (localhost:1933) 是工具执行层，不是 LLM Provider
- MiniMax-M2.7 做 AI 推理（意图解析、摘要生成）
- OpenViking MCP Server 做工具调用（文件操作、代码执行）
- MCP Client (`mcp_client.py`) 通过 stdio 与 OpenViking 通信，集成在 Service Layer
- Trade-off: 工具执行与推理解耦，增加 ~100-300ms 延迟，但架构更清晰

## 技术评审结论 (2026-05-20)
- 架构基本合理，无需推翻重写
- 修正 Phase 边界：Channel Adapter 归入 Phase 3，精力曲线/冲突检测 MVP 不做
- 技术栈：Python Agent + FastAPI + SQLite(MVP) / PostgreSQL(正式) + Vue 3 + MiniMax-M2.7 + OpenViking MCP
- 架构层面主要风险：微信 API 限制（小厂个人测试号）、SQLite 并发（单用户够用）
