## 一、产品愿景与核心价值

**一句话定位**: 微信里的个人 AI 执行力教练——用微信消息创建任务，用 AI 规划时间，用多角色视角切换工作与生活。

**核心差异化**:
1. **微信即终端** — 不需要打开任何 App，微信消息直接创建任务
2. **多角色标签** — 同一人的 Team Lead / Developer / PM 身份有独立任务池，视角切换无缝
3. **AI 原生规划** — 每日自动生成时间块计划，推送微信

**目标用户**: 身兼多职的技术管理者（Team Lead + Committer + 项目负责人），每天处理跨角色任务流，需要快速捕获和智能规划。

**市场定位确认**: 经竞品扫描（Todoist、飞书、钉钉、Mem、Obsidian 等），个人多角色 AI Agent + 微信集成是明确市场空白，尚无直接竞品。窗口期仍在。

## 二、MVP 定义（v0.1 — 最小闭环）

### 2.1 MVP 目标

**唯一目标**: 用户发微信 `/任务 xxx` → AI 解析 → 任务创建 → Web 端可查看 → 任务完成 → 微信推送确认。

### 2.2 MVP 必须有的功能

| 功能 | 描述 | 理由 |
|------|------|------|
| 角色管理 CRUD | 创建/切换多个角色标签 | 多角色的基础，没有这个其他都无法落地 |
| 任务管理 CRUD | 创建/查看/完成/删除任务 | 核心闭环的载体 |
| 微信消息 → 任务 | 接收微信文字消息，AI 解析创建任务 | MVP 的核心差异化 |
| 任务 → 微信通知 | 任务创建/分配/完成时推送微信 | 让用户感知到闭环 |
| AI 意图解析 | 自然语言提取：标题、优先级、预估时长 | 微信→任务的智能化核心 |
| Web 端任务列表 | 基础列表视图，支持按角色筛选 | 用户查看任务的入口 |

### 2.3 MVP 明确不做的功能

以下功能在 architecture.md 中有设计，但 **MVP 阶段不实现**，理由充分：

- ❌ 每日 AI 规划（需要更多 AI 调优才能可靠）
- ❌ 冲突检测与优先级推荐（需要真实使用数据才能校准）
- ❌ 精力曲线分析（Phase 2/3 考虑，MVP 不做）
- ❌ 外部系统集成（Linear/GitHub/飞书）→ 归入 Phase 3
- ❌ PWA / 微信内嵌浏览器适配（Web 先做，微信端先靠微信推送）
- ❌ 定时每日摘要推送（需要稳定运行后再加）
- ❌ 语音消息解析
- ❌ 数据分析统计

### 2.4 MVP 成功标准

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| 配置完成时间 | 用户 5 分钟内完成初始配置并能发送微信创建任务 | 内测人工计时 |
| 微信→任务转化率 | AI 正确解析消息比例 > 90% | 日志统计 |
| 消息处理延迟 | P99 < 5s（AI 解析时间） | 服务端埋点 |
| 连续活跃 | 用户连续 3 天使用微信创建任务且回访 Web 端 | 事件埋点 |
| **意图识别准确率** | > 90%（AI 正确理解用户意图的比例） | Golden Dataset 验证 |
| **任务创建成功率** | > 95%（解析成功且成功写入 DB 的比例） | 服务端埋点 |
| **误识别率** | < 5%（错误解析导致创建错误任务的比率） | 日志抽样审计 |
| **P95 消息处理延迟** | < 3s（比 P99 更严格的延迟保障） | 服务端埋点 |

## 三、产品路线图（三个阶段）

### Phase 1: MVP（v0.1）— 核心闭环

**周期**: 4 周
**目标**: 跑通微信 ↔ 任务 ↔ AI 解析的最小闭环

#### 交付物

- [ ] 角色 CRUD + Web 端切换
- [ ] 任务 CRUD + 状态流转（Inbox → Todo → Done）
- [ ] 微信消息接收 Webhook（微信客服消息，无须企业号）
- [ ] AI 意图解析（自然语言 → 结构化任务字段）
- [ ] 基础微信推送（任务创建通知）
- [ ] Web 端任务列表（按角色筛选）

#### 技术依赖

- 微信客服消息测试号申请（推荐，免费，无需企业主体）
- MiniMax API Key 配置
- SQLite 初始化

#### 关键指标

- 微信→任务转化率 > 90%（AI 能正确解析的消息比例）
- P99 < 5s，P95 < 3s
- 意图识别准确率 > 90%，误识别率 < 5%
- 任务创建成功率 > 95%

#### 技术评审确认（Phase 边界）

- Channel Adapter 抽象层：**归入 Phase 3**，MVP 专注微信单一通道
- 精力曲线分析：**不列入 MVP**，Phase 2/3 考虑
- 冲突检测与优先级推荐：**不列入 MVP**，需要真实数据校准
- WeChat crypto.py：**安全关键路径**，需 TDD 先行（见测试策略）

---

### Phase 2: 智能化（v0.2）— AI 规划能力

**周期**: 4 周
**前置条件**: MVP 稳定运行，真实用户在用

#### 交付物

- [ ] **AI 每日时间块规划** — 自动生成次日时间表，推送微信确认
- [ ] **AI 角色自动推断** — 根据任务内容推荐归属角色
- [ ] **AI 优先级推荐** — 结合截止日期 + 工作量 + 角色给出优先级
- [ ] **AI 任务摘要** — 长任务自动生成简短摘要
- [ ] **增量重规划** — 任务完成/取消时自动调整当天计划
- [ ] **每日摘要定时推送** — 早 8:30 推送当日计划

#### AI 质量体系（Phase 2 建立）

| 指标 | 目标 | 说明 |
|------|------|------|
| 意图识别准确率 | > 92% | Golden Dataset 持续验证 |
| 摘要生成质量评分 | > 4/5（人工评估） | 抽样评估 |
| 规划采纳率 | > 60% | 用户接受 AI 规划的比例 |
| Provider 降级触发率 | < 2% | MiniMax 不可用时降级 DeepSeek |

#### 功能优先级说明

为什么不把 AI 规划放在 MVP：
1. AI 规划的输出质量依赖真实任务数据，没有数据就无法校准 prompt
2. 用户需要先建立「任务录入」习惯，规划才有意义
3. MVP 阶段让用户用起来，同时收集真实数据，为 AI 规划提供样本

---

### Phase 3: 扩展与生态（v1.0）— 外部集成 + 多用户

**周期**: 6 周
**前置条件**: Phase 2 验证了 AI 规划价值

#### 交付物

- [ ] **Linear 集成** — 双向同步 Engineering 任务
- [ ] **GitHub 集成** — Issue/PR 关联
- [ ] **飞书通知** — 备用推送渠道
- [ ] **Channel Adapter 重构** — 抽象消息通道层，支持多渠道接入
- [ ] **PostgreSQL 迁移** — SQLite → PostgreSQL（多用户支持）
- [ ] **Redis 缓存** — AI 响应缓存 + 通知队列
- [ ] **PWA 适配** — 微信浏览器离线支持
- [ ] **数据分析面板** — 角色时间分配、周报生成

#### 功能优先级说明

- 多用户是 Phase 3 才考虑的事：个人使用足够简单，不需要账号体系；引入多用户会让 SQLite 的局限提前暴露
- Channel Adapter 抽象层在 Phase 3 再构建：MVP 阶段专注微信单一通道，Phase 2 验证核心价值后再扩展
- 微信小程序端暂不考虑，Web + 微信推送已覆盖 90% 场景

#### 关键指标

- 外部系统同步成功率 > 95%
- 多用户并发写入无数据丢失（PostgreSQL 迁移验证）

## 四、非功能需求

### 4.1 性能目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 微信消息 → 任务响应 | < 5s (P99), < 3s (P95) | AI 解析时间 |
| Web 端页面加载 | < 1s | 首屏加载 |
| 并发用户（SQLite 阶段） | ≤ 5 | 单用户 + 少量并发读 |
| AI 错误率 | < 5% | 解析失败时的优雅降级 |

### 4.2 AI 质量目标（v0.2 建立）

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 意图识别准确率 | > 90% (MVP), > 92% (v0.2) | Golden Dataset 验证 |
| 任务创建成功率 | > 95% | 服务端埋点 |
| 误识别率 | < 5% | 日志抽样审计 |
| AI 摘要质量 | > 4/5 | 人工抽样评估 |
| Provider 降级触发率 | < 2% | MiniMax → DeepSeek 自动降级 |

### 4.3 部署目标

| 环境 | 部署方式 | 说明 |
|------|----------|------|
| 开发 | 本地 `uvicorn` | 快速迭代 |
| 生产 | Docker Compose | PostgreSQL + Redis + App |

### 4.4 安全目标

- JWT Token 认证
- 微信 Webhook 签名验证（防止伪造消息）
- 配置文件中不含明文密钥（敏感信息走环境变量或密钥服务）
- **WeChat crypto.py 安全关键路径**：需 TDD 先行，发布前通过安全审计

### 4.5 测试覆盖率目标

| 层级 | 目标 | 说明 |
|------|------|------|
| 单元测试 | 核心业务逻辑 > 80% | crypto.py、task/role CRUD |
| 集成测试 | API 端点 > 90% | FastAPI 路由全覆盖 |
| E2E 测试 | 核心路径覆盖 | Playwright Web E2E |
| AI 质量测试 | Golden Dataset + Schema 验证 | 意图解析质量监控 |

## 五、开放产品决策（已确定项）

| # | 问题 | 最终决策 | 说明 |
|---|------|----------|------|
| 1 | **微信方案** | 微信客服消息（免费，无须企业号） | Phase 1 接入；正式版可升级企业微信 |
| 2 | **多用户时机** | Phase 3 之前不考虑 | 个人 MVP 先验证核心价值 |
| 3 | **定价模式** | MVP 完全免费，v0.2+ 再定 | 参考钉钉 ¥99/年、Todoist $5/月 |
| 4 | **开源策略** | 全开源（MIT） | 快速建立社区，Phase 3 后实施 |
| 5 | **LLM Provider** | MiniMax 默认 + DeepSeek 降级 | 成本最优，Provider 降级自动化 |

## 七、风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 微信 API 限制导致推送不稳定 | 高 | 微信客服消息作为降级（免费，无需企业号）；正式版走企业微信 |
| AI 解析质量不佳（意图识别失败） | 中 | MVP 阶段保留手动修正入口；Golden Dataset 持续优化 prompt |
| 用户不愿在微信外打开 Web | 中 | 微信推送中包含任务摘要，减少打开 Web 的必要性 |
| SQLite 并发写入瓶颈 | 低（单用户） | Phase 3 迁移 PostgreSQL |
| **MiniMax API 不可用** | 中 | 自动降级 DeepSeek；置信度 < 0.7 时标记人工确认 |
| **WeChat crypto 安全漏洞** | 高 | TDD 先行，发布前安全审计 |

## 八、产品原则（团队共识）

1. **先跑通，再优化** — 不为 5% 的场景牺牲 95% 的体验
2. **微信是入口，不是终点** — 所有核心操作在 Web 端都能完成
3. **AI 是助理，不是主角** — 用户始终有最终决策权
4. **数据留在本地** — 不上云不需要考虑数据主权问题
5. **公开讨论，私下纠正** — 产品决策有据可查，错了改就是
6. **AI 质量可测量** — 意图识别准确率、任务创建成功率必须可量化
